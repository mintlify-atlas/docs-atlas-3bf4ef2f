## What bidirectional means

A traditional router only goes in one direction: it takes a URL and produces an application-level value. Generating a URL from an application value is handled separately — usually by building strings manually or with a template system.

swift-url-routing is **bidirectional**: the same router object that parses a URL into an `AppRoute` can also turn an `AppRoute` back into a URL. There is a single source of truth for what your URLs look like, defined once in the router.

  <Card title="Parsing" icon="arrow-right">
    `URL` → `AppRoute`

    Consumes an incoming request and extracts structured data.
  </Card>
  <Card title="Printing" icon="arrow-left">
    `AppRoute` → `URL`

    Reconstructs a valid URL from a structured route value.
  </Card>
</CardGroup>

## Why this matters

When URL structure is defined in only one place, you get several practical benefits:

- Renaming a URL path segment — say `/books` to `/library` — requires one change, in the router. Every link generated from that router automatically uses the new path.
- Links generated by the router are always valid. If you add a required field to a route case, the compiler will flag every callsite that constructs that case, including the ones that generate links.
- There is no separate link-building layer to keep in sync with the routing layer.

<Note>
This is especially valuable in server-side applications that both parse incoming requests and render HTML links. With a bidirectional router, link generation is guaranteed to be consistent with request routing.
</Note>

## The `ParserPrinter` protocol

The `ParserPrinter` protocol is the foundation of bidirectionality. A type that conforms to `ParserPrinter` can both:

- **Parse**: consume a `URLRequestData` and produce an output value.
- **Print**: take an output value and produce a `URLRequestData`.

Every building block in swift-url-routing — `Route`, `Path`, `Query`, `Field`, and so on — conforms to `ParserPrinter` when its components do. Because these pieces compose, the assembled router is automatically a `ParserPrinter` too.

## Parsing: URL to route

The library extends any parser whose input is `URLRequestData` with three matching methods:

```swift
// From a full URL
try appRouter.match(url: url)

// From a path string (including query)
try appRouter.match(path: "/books/search?query=Blob")

// From a URLRequest (includes method, headers, body)
try appRouter.match(request: urlRequest)
```

All three convert the incoming value into `URLRequestData`, then call `parse` on the router. If no route matches, a `RoutingError` is thrown.

A complete deep-link handler looks like this:

```swift
func handleDeepLink(url: URL) throws {
  switch try appRouter.match(url: url) {
  case .books:
    // navigate to books screen

  case let .book(id: id):
    // navigate to book with id

  case let .searchBooks(query: query, count: count):
    // navigate to search screen with query and count
  }
}
```

## Printing: route to URL

The library extends any `ParserPrinter` whose input is `URLRequestData` with three printing methods:

```swift
// Produce a path string (with query parameters)
appRouter.path(for: .searchBooks(query: "Blob Bio"))
// "/books/search?query=Blob%20Bio"

// Produce a full URL
appRouter.url(for: .book(id: 42))
// URL("..."), relative without a base

// Produce a URLRequest
try appRouter.request(for: .books)
```

These are particularly useful for generating links in server-rendered HTML:

```swift
ul {
  for book in books {
    li {
      a {
        book.title
      }
      .href(appRouter.path(for: .book(id: book.id)))
    }
  }
}
```

Which produces:

```html
<ul>
  <li><a href="/books/1">Blob Autobiography</a></li>
  <li><a href="/books/2">Blobbed around the world</a></li>
  <li><a href="/books/3">Blob's guide to success</a></li>
</ul>
```

<Tip>
`url(for:)` never throws. If the router cannot print the given route, it returns a `#route-not-found` URL and triggers a runtime breakpoint in debug builds so you catch the misconfiguration early.
</Tip>

## Prepending a base URL

When printing absolute URLs, use `baseURL(_:)` to prepend a scheme, host, and optional path prefix:

```swift
let apiRouter = appRouter.baseURL("https://api.example.com/v1")

apiRouter.url(for: .book(id: 42))
// https://api.example.com/v1/books/42
```

`baseURL` is non-destructive: it only fills in URL components that the route did not already specify. You can also supply arbitrary default request data — for example, to attach authentication headers to every printed request:

```swift
let authenticatedRouter = appRouter
  .baseRequestData(.init(headers: ["Authorization": ["Bearer token"]]))
```

## Traditional vs. bidirectional routers

<Tabs>
  <Tab title="Traditional (one-way)">
    ```swift
    // Routing: defined in one place
    router.get("/books/:id") { req in
      let id = req.parameters.get("id", as: Int.self)!
      return fetchBook(id: id)
    }

    // Link generation: defined separately, duplicated knowledge
    func bookURL(id: Int) -> String {
      "/books/\(id)"  // must stay in sync manually
    }
    ```

    A typo in `bookURL` produces a broken link that only fails at runtime.
  </Tab>
  <Tab title="Bidirectional (swift-url-routing)">
    ```swift
    // Routing and link generation: single definition
    let appRouter = OneOf {
      Route(.case(AppRoute.book(id:))) {
        Path { "books"; Digits() }
      }
      // ...
    }

    // Parsing
    try appRouter.match(url: url)

    // Printing — same router, no duplication
    appRouter.path(for: .book(id: 42))
    // "/books/42"
    ```

    The URL structure is defined once. Changing it updates both parsing and link generation automatically.
  </Tab>
</Tabs>
